启动引擎

Claude Code 的启动设计精妙绝伦 —— 一键启动，0.1 秒内就能响应最简单的命令

生活化比喻
想象你问一个人"你叫什么名字？"——他不需要翻出身份证、查户口本，张口就答。Claude Code 的启动也是这样：如果你只是问 --version，它连重型模块都不加载，直接回答。这叫做 「快速路径」(Fast Path)。

🏁 快速路径设计

这段代码展示了 Claude Code 启动时的第一个判断。它的核心理念是：能不加载的模块就不加载。

设计精髓：当用户只想看版本号时，整个程序只执行 3 行代码就退出了。不会加载上千个模块、不会初始化数据库、不会连接网络。

后面的代码也是类似思路：不同的命令走不同的"快车道"，每条路只加载自己需要的模块。这就像高速公路的分流匝道——去机场的车不需要经过市区。

新手要记住的原则：「延迟加载」(Lazy Loading) —— 只在真正需要时才加载代码，是提升性能的黄金法则。
src/entrypoints/cli.tsx TypeScript
// ============================================ // Claude Code 的启动入口 —— 一切从这里开始 // ============================================ async function main(): Promise<void> { // 第一步：拿到用户在命令行输入的参数 const args = process.argv.slice(2) // 🏎️ 【快速路径】如果用户只想看版本号 // 直接打印，立刻退出！ // 不加载任何其他模块 —— 极速响应 if ( args.length === 1 && (args[0] === '--version' || args[0] === '-v') ) { console.log(`${MACRO.VERSION} (Claude Code)`) return // 👋 用完即走 } // 只有走到这里，才开始加载性能追踪器 // 注意：用的是 await import() 动态导入 const { profileCheckpoint } = await import('../utils/startupProfiler.js') profileCheckpoint('cli_entry') // 接下来根据不同参数，走不同的"快车道" // 守护进程？桥接模式？模板生成？各走各路 // ... } main()

⚡ 并行预加载

当 Claude Code 真正需要完整启动时，它会同时做好几件事，而不是排队一件一件做。

核心技巧：在 JavaScript/TypeScript 中，Promise 可以让多个异步操作同时进行。就像你一边烧水、一边切菜、一边热锅——而不是先烧水等它开了再切菜。

这段初始化代码在模块还没加载完的时候，就已经开始读取系统配置和钥匙串（密码管理器）了。等模块加载完毕时，配置数据也差不多准备好了，省去了等待时间。

// ============================================ // 初始化阶段 —— 多条线同时开工 // ============================================ // 🏃 在模块还没加载完时，先启动这些"预热"任务 // 就像餐厅开门前：水已经在烧、面已经在揉 // 任务1：读取系统管理配置（MDM 策略） prefetchMDMRawRead() // ⚡ 不等结果，先发出去 // 任务2：预取钥匙串中的凭证 prefetchKeychain() // ⚡ 也不等，同时进行 // 任务3：验证 settings.json 配置 enableConfigs() // 任务4：仅应用"安全"的环境变量 // 注意：信任对话框出现之前， // 不能让项目配置影响环境 —— 这是安全考量 applyOnlySafeEnvVars() // 任务5：后台异步填充 OAuth、远程设置等 // 这些不会阻塞用户交互 await Promise.all([ populateOAuth(), fetchRemoteSettings(), loadPolicyLimits(), ])

消息循环 · 对话的心脏

QueryEngine 是整个 Claude Code 的核心引擎 —— 每一次对话都由它驱动

生活化比喻
消息循环就像一条寿司传送带。你（用户）把需求写在纸条上放上传送带，传送带把它送到厨房（Claude API），厨师做好寿司（响应）一盘一盘放回传送带（流式输出），你一盘一盘地拿（看到回答逐字出现）。如果厨师需要去仓库拿食材（调用工具），传送带会先绕一圈仓库，拿到食材后继续做寿司。

QueryEngine 消息循环详解

📝 用户发送消息
↓
processUserInput() — 解析斜杠命令、附件
↓
💬 推入 mutableMessages 数组（先保存到磁盘，确保可恢复！）
↓
🤖 调用 Claude API（AsyncGenerator 流式返回）
↓
💬 普通文本回复
直接渲染到终端
🔧 工具调用请求
执行工具 → 结果送回 API
↓
✔️ 一轮对话结束（或继续下一轮工具调用）

🎣 AsyncGenerator — 流式魔法

这是 Claude Code 最核心的设计之一。submitMessage 不是等 AI 完全回答完再一次性返回，而是像水龙头一样，来一滴返回一滴。

什么是 AsyncGenerator？ 想象一个水龙头。普通函数是一桶水——要等装满了才能用。AsyncGenerator 是自来水管——打开就有水流出来，随来随用。

关键字 async * 和 yield 让这个函数可以一边从 API 接收数据，一边把数据交给 UI 渲染。用户看到的就是文字一个字一个字地出现。

妙处在于：mutableMessages 是一个共享的可变数组。消息在发给 API 之前就已经保存到磁盘了——即使程序意外崩溃，下次启动也能恢复对话！
src/QueryEngine.ts TypeScript
// ============================================ // QueryEngine —— 整个 Claude Code 的心脏 // ============================================ export class QueryEngine { // 📦 共享消息存储 —— 跨轮次保持对话历史 // 就像聊天记录一样，所有消息都存在这里 mutableMessages: Message[] // 🛑 中止控制器 —— 可以随时喊停 abortController: AbortController // ❌ 记录每次权限被拒绝的情况 permissionDenials: PermissionDenial[] // 🎣 核心方法：提交一条消息 // 注意 async * —— 这是 AsyncGenerator！ // 它不会等回答完毕，而是边收边返 async *submitMessage( prompt: string, options?: { uuid?: string } ): AsyncGenerator<SDKMessage> { // 第一步：处理用户输入 // 识别斜杠命令、附件、特殊指令 const { messages, shouldQuery } = await processUserInput({ input: prompt }) // 💾 先保存！再调 API // 即使 API 调用中断，消息也不会丢 this.mutableMessages.push(...messages) // 第二步：组装系统提示词 const { defaultSystemPrompt, userContext } = await fetchSystemPromptParts({...}) // 第三步：调用 API 并流式返回结果 // yield* 把 query() 的每一条消息都转发出去 yield* query({ messages: this.mutableMessages, systemPrompt, canUseTool: wrappedCanUseTool, }) } }

工具系统 · 万能工具箱

Claude Code 有 40+ 种工具，但它们共享同一套优雅的接口设计

生活化比喻
工具系统就像一个标准化的工具箱。无论是螺丝刀、锤子还是电钻，它们的手柄都是统一规格的——你只需要学会握手柄的方式，就能使用任何工具。在 Claude Code 中，这个"统一的手柄"就是 Tool 接口。每个工具都实现了相同的方法：call（使用工具）、description（说明书）、inputSchema（需要什么原料）。

🏗️ Tool 接口 — 万物之母

这是 Claude Code 中最重要的类型定义之一。所有工具——编辑文件、执行命令、搜索代码——都必须实现这个接口。

为什么这样设计？ 统一接口让系统可以一视同仁地对待所有工具。添加新工具只需要实现这几个方法，不需要改动系统其他部分。这就是面向对象设计中的 「开闭原则」：对扩展开放，对修改关闭。

特别注意这些巧妙的属性：

  • isConcurrencySafe — 这个工具能否和别的工具同时执行？
  • isReadOnly — 只读操作不需要权限确认
  • shouldDefer — 大型工具延迟加载，节省启动时间
  • aliases — 工具改名后旧名字仍然可用
架构启示：好的接口设计让每个工具都能「自我描述」——系统不需要知道工具的具体实现，只需要问它几个问题就能做出正确的调度决策。
src/Tool.ts TypeScript
// ============================================ // Tool 接口 —— 所有工具的"标准化手柄" // ============================================ export type Tool<Input, Output> = { // 📛 别名：工具改名后旧名字仍能用 // 就像手机号码变了，但旧号码还能转接 aliases?: string[] // 🔍 搜索提示词：帮 AI 找到正确的工具 searchHint?: string // 🎯 核心方法：真正执行工具的地方 call( args: Input, // 输入参数 context: Context, // 运行环境 canUseTool: Fn, // 权限检查函数 ): Promise<ToolResult> // 📄 动态描述：根据输入内容生成说明 description(input: Input): Promise<string> // 📋 输入校验规则（Zod Schema） readonly inputSchema: ZodSchema // === 智能调度属性 === // 🤝 能否与其他工具并行执行？ // 例如：两个 Read 可以同时跑， // 但两个 Edit 不行（可能改同一个文件） isConcurrencySafe(input: Input): boolean // 🔒 是否只读？只读操作可以跳过权限确认 isReadOnly(input: Input): boolean // 💣 是否有破坏性？破坏性操作需要用户确认 isDestructive?(input: Input): boolean // ⏳ 是否延迟加载？ // 大型工具的 Schema 很大， // 延迟到真正使用时才加载，加速启动 shouldDefer?: boolean }

🚪 特性门控 — 编译时裁剪

Claude Code 有很多内部专用功能。它用 Feature Gate 在编译时就把不需要的代码删掉，而不是运行时判断。

什么是「死代码消除」(DCE)? 想象你出版一本书。内部版有所有章节，公开版只保留公开章节。传统做法是把删掉的章节用墨水涂黑（运行时判断），但 Claude Code 的做法是直接不印那些页（编译时消除）。最终产物更小、更快、更安全。

feature() 是一个编译时函数。当编译器看到 feature('PROACTIVE') === false 时，整个 if 分支和相关的 require 都会被删除——最终的发布版本中根本不存在这些代码。

权限守卫 · 八重安检门

每个工具调用都要过「八重门」—— 确保 AI 永远不会做出超出授权的事情

生活化比喻
权限系统就像机场安检。你的行李（工具调用）要过好几道关卡：先是身份验证（规则匹配），然后 X 光机扫描（AI 分类器判断安全性），可疑物品要开箱检查（用户确认弹窗），危险品直接扣留（拒绝执行）。最巧妙的是：如果 AI 分类器连续拒绝太多次，系统会自动切换到"人工检查"模式——因为可能是分类器出了问题。

权限决策树 —— 一个工具调用要过几道关？

↓

↓

↓ 如果结果是 "ask"
✅ 自动模式
AI 分类器快速判断
💬 交互模式

④ 拒绝追踪 —— 连续拒绝过多次？切换到人工确认模式

🔍 规则匹配引擎

权限规则是数据驱动的——不是在代码里硬写 if/else，而是把规则当作数据来处理。

规则匹配的优先级：
1. deny（拒绝）> 2. ask（询问）> 3. allow（允许）
安全性优先！deny 规则永远赢。

规则支持通配符匹配，比如 Bash(prefix:git *) 表示"允许所有以 git 开头的命令"。这种声明式的规则系统让权限配置变得像写配置文件一样简单。

精妙之处：拒绝追踪机制会记录连续被拒绝的次数。如果 AI 分类器连续拒了 N 次，系统认为"分类器可能搞错了"，会自动退回到让用户手动确认——这是一种自我纠错的闭环！
src/utils/permissions/permissions.ts TypeScript
// ============================================ // 权限决策主函数 —— 八重安检的入口 // ============================================ export const hasPermissionsToUseTool = async ( tool, // 要检查的工具 input, // 工具的输入参数 context, // 运行上下文 ): Promise<PermissionDecision> => { // 第一道：核心权限检查 // 包含规则匹配、Hook 检查、分类器 const result = await hasPermissionsToUseToolInner(tool, input, context) // 第二道：根据权限模式转换结果 // - Auto 模式 → 让 AI 分类器决定 // - DontAsk 模式 → 把"询问"变成"拒绝" // - Plan 模式 → 需要方案审批 // 📊 第三道：拒绝追踪（自我纠错闭环！） if (result.behavior !== 'allow') { // 记录这次拒绝 recordDenial(denialState, ...) // 🤔 连续拒绝太多次了？ // 可能是分类器出了问题！ // 自动切换到"人工确认"模式 if (shouldFallbackToPrompting(denialState)) { return { behavior: 'ask' } // ↑ 不再自动拒绝，改为问用户 } } return result }

记忆宫殿 · 长期记忆系统

Claude Code 如何记住你是谁、你喜欢什么、上次聊到哪里

生活化比喻
记忆系统就像一座图书馆。最底层的地下室放着"馆规"（系统管理配置），一楼放着你的"个人日记"（用户全局配置），二楼放着"项目手册"（项目级 CLAUDE.md），三楼放着"条件索引卡"（只在特定文件被编辑时才激活的规则），顶层的阁楼放着"AI 自己的笔记本"（自动记忆）。每次对话开始，图书管理员会按顺序把所有相关的书拿出来，摊在桌上给 AI 看。

📚 @include 指令 — 记忆的"超链接"

Claude Code 的记忆文件支持互相引用！就像维基百科的超链接，一个记忆文件可以"导入"另一个文件的内容。

为什么这很重要？你可以把公共规范放在一个文件里，然后在多个项目的 CLAUDE.md 中引用它。改一处，所有项目自动更新。这就是编程中"不要重复自己"（DRY 原则）的体现。

系统还内置了循环引用保护：最多递归 5 层，防止 A 引用 B、B 又引用 A 导致无限循环。

条件规则更酷：.claude/rules/ 目录下的 Markdown 文件可以设置 paths: 前缀，比如 paths: src/**/*.ts。只有当你编辑匹配这个模式的文件时，这条规则才会生效！就像"只有打开保险箱时才亮起的警告灯"。
src/utils/claudemd.ts TypeScript
// ============================================ // @include 指令 —— 记忆文件的"超链接"系统 // ============================================ // 🔗 从 Markdown 文本中提取 @include 路径 // 支持三种路径格式： // @./relative-path.md (相对路径) // @~/home-path.md (主目录) // @/absolute-path.md (绝对路径) const MAX_INCLUDE_DEPTH = 5 // 🛡 防止循环引用：A引用B、B引用A // 最多递归 5 层就停止 // ============================================ // 条件规则匹配 —— 只在特定场景激活 // ============================================ export async function processConditionedMdRules( targetPath: string, // 正在编辑的文件 rulesDir: string, // .claude/rules/ 目录 ): Promise<MemoryFileInfo[]> { // 📁 读取 rules 目录下所有 .md 文件 const ruleFiles = await readDir(rulesDir) for (const file of ruleFiles) { // 🔍 解析 frontmatter 中的 paths 字段 // 例如: paths: src/**/*.ts const { paths } = parseFrontmatter(file) // 🎯 用 glob 模式匹配当前编辑的文件 if (matchesGlob(targetPath, paths)) { // ✅ 匹配成功！把这条规则加载到上下文中 results.push(ruleContent) } // ❌ 不匹配？跳过，节省 token } return results }

📦 上下文压缩 — 图书管理员整理书架

AI 的记忆窗口是有限的（比如 200K token）。当对话太长时，Claude Code 会启动自动压缩。

压缩的原则：不是简单地删掉旧消息！而是让 AI 先写一份"会议纪要"，总结之前的内容。这份纪要保留了所有重要信息：用户的需求、已修改的文件、遇到的错误和解决方案。

更精妙的是工具结果清除策略：当 token 超过阈值时，系统会清除旧的工具调用结果（比如大量的搜索结果），但保留文件编辑记录——因为编辑记录是"承重墙"，删掉会影响 AI 理解代码的当前状态。

💡 缓存分界策略 — 系统提示词的巧妙切割

系统提示词被一条"分界线"(SYSTEM_PROMPT_DYNAMIC_BOUNDARY) 切成两半：

分界线之前：通用指令（"你是 Claude Code"、"如何使用工具"）—— 这部分对所有用户都一样，可以全局缓存，所有人共用一份。

分界线之后：个性化内容（你的 CLAUDE.md、记忆文件、环境信息）—— 每个人不同，不能缓存。

这一刀切得精妙绝伦：全局缓存意味着 Anthropic 的服务器只需要处理一次通用指令，之后所有用户的请求都能复用——大幅降低延迟和成本。

🎣 Hook 执行 — 异步生成器管道

Hook 的执行结果不是一个简单的 true/false，而是一个流式管道——每个 Hook 可以产出多种类型的结果。

Hook 可以做的事：
• 返回附加消息（让 AI 知道额外信息）
• 阻止工具执行（exit code 2）
• 修改工具输出
• 注入额外上下文
• 阻止对话继续
自我纠错的闭环：最精彩的 Hook 是 TeammateIdle——当一个 AI 子代理准备"下班"时，这个 Hook 会检查任务队列里还有没有未完成的工作。如果有，Hook 返回 exit code 2，阻止它休息，让它继续干活！这就是"自驱动"闭环。

🤖 Agent 子代理 — 分身术

Claude Code 可以生成子代理来并行处理任务。主代理就像一个项目经理，把任务分配给多个"分身"同时执行。

🎮 Blit 优化 — 游戏引擎的智慧

Blit（Block Image Transfer，块图像传输）是游戏引擎中常见的渲染优化技术。Claude Code 把它搬到了终端渲染中。

核心原理：如果一个组件的位置和大小都没变，就直接把上一帧的那个区域"复印"过来，整个子树的渲染都跳过！

这段代码是渲染引擎的核心优化点。nodeCache 记录了每个组件在上一帧的位置和大小。如果位置没变，就调用 blit() 直接复制，跳过整个子树。

实际效果：当 AI 正在流式输出文字时，屏幕上只有最后几行在变化。上面的内容全部通过 Blit 复制——工作量从 O(屏幕面积) 降低到 O(变化的单元格数)。

// ============================================ // Blit 优化 —— 从游戏引擎借来的智慧 // ============================================ // 🔍 从缓存中查找这个节点上一帧的位置 const cached = nodeCache.get(node) // ❓ 这个节点没有变化？ // 位置和大小都和上一帧一样？ if ( !node.dirty && // 内容没变 cached && // 有缓存记录 cached.x === x && // X 坐标没变 cached.y === y && // Y 坐标没变 cached.width === width && cached.height === height && prevScreen // 有上一帧的数据 ) { // 🎮 Blit! 直接"复印"上一帧的这块区域 // 整个子树的渲染全部跳过！ output.blit(prevScreen, fx, fy, fw, fh) return // ⚡ 提前返回，省去大量计算 } // 如果位置变了，才需要重新渲染这个节点 // ... // ============================================ // 双缓冲机制 —— 两张画布轮流使用 // ============================================ // // frontFrame = 当前显示的画面 // backFrame = 正在绘制的下一帧 // // 画完后交换： // backFrame = frontFrame (旧画布回收) // frontFrame = newFrame (新画布上屏) // // 好处：不需要每帧都分配新的内存

📦 打包单元格 — 极致的内存优化

终端上的每个字符都需要记录：字符内容、样式（颜色/粗体）、宽度、超链接。Claude Code 把这些信息压缩成两个 32 位整数。

为什么用 TypedArray?
• 一个单元格 = 8 字节 = CPU 一个缓存行能放 8 个
• 批量复制用 TypedArray.set()，比逐个赋值快 10 倍
• 位操作 styleId & 1 就能判断空格是否可见

还有一个巧妙的字符池和样式池设计：常见字符（ASCII）通过数组直接查找（O(1)），样式转换（从样式 A 切换到样式 B 的 ANSI 代码）被预计算并缓存——渲染时零内存分配！

安全堡垒 · 纵深防御

从路径攻击防御到引号智能保留 —— Claude Code 的安全设计令人叹为观止

生活化比喻
安全系统就像一座中世纪城堡。外面有护城河（路径规范化检查），城门有卫兵（危险文件清单），城墙上有弓箭手（符号链接解析），城内还有层层关卡（权限决策树）。即使有人偷偷绕过一道防线，下一道防线也会把它挡住——这叫「纵深防御」(Defense in Depth)。

🚫 危险文件保护

.bashrc、.gitconfig、.zshrc 等文件被列入黑名单。这些文件可以在系统启动时执行代码，修改它们相当于打开了后门。即使用户主动授权，也会被额外警告。

🔍 Windows 路径攻击防御

检测 NTFS 备用数据流 (file.txt::$DATA)、8.3 短名称 (GIT~1)、DOS 设备名 (.git.CON) 等 Windows 特有的路径伪装技巧。防止通过路径花招绕过安全检查。

🔗 符号链接追踪

检查文件时，不仅检查文件本身，还追踪整条符号链接链。即使 safe-file 是一个指向 /etc/passwd 的链接，也会被发现和拦截。

🔠 大小写不敏感保护

在 macOS 和 Windows 上，.Claude 和 .claude 是同一个目录。所有路径比较都先转成小写，防止通过大小写混写绕过检查。

✍️ 智能引号保留

如果文件里用的是弯引号 (“”)，编辑后自动保持弯引号风格。甚至能区分缩写中的撇号 (don’t) 和收尾引号——用 Unicode 正则匹配字母来判断。

⏰ 文件陈旧检测

编辑文件前检查：文件是否在你读取之后被其他程序修改了？Windows 上时间戳可能被云同步软件篡改，所以额外做内容比对来防止误判。

🛡️ 写入权限决策树

这是安全系统最核心的代码之一 —— 文件写入操作要过8 道检查。

关键设计：安全检查在 allow 规则之前！
即使用户配置了宽泛的 allow 规则（比如"允许编辑所有文件"），危险文件仍然会被拦截。这防止了"好心办坏事"的情况。

注意第 4 步和第 5 步的顺序。在大多数系统中，allow 规则会直接放行。但 Claude Code 先做安全检查，再看 allow 规则。这是一个精妙的设计决策——把安全性放在便利性之上。

src/utils/permissions/filesystem.ts TypeScript
// ============================================ // 文件写入权限 —— 8 道安检门 // ============================================ // ① 显式拒绝规则 —— 黑名单直接拦截 if (matchesDenyRule(path)) return 'deny' // ② 内部可编辑路径 —— 计划文件、草稿本 if (isInternalEditable(path)) return 'allow' // ③ Claude 文件夹会话规则 if (isClaudeFolderSessionRule(path)) return 'allow' // ④ ⚠️ 安全检查 —— 在 allow 之前！ // 即使用户配置了宽泛的 allow 规则， // 危险文件仍然会被拦截 if (isDangerousFile(path)) { return 'ask' // 强制询问用户确认 } // ⑤ 显式询问规则 if (matchesAskRule(path)) return 'ask' // ⑥ 工作目录自动允许 // 如果是 acceptEdits 模式 if (isInWorkingDir(path) && mode === 'acceptEdits') return 'allow' // ⑦ 显式允许规则 if (matchesAllowRule(path)) return 'allow' // ⑧ 默认 —— 没有任何规则匹配？ // 安全起见，询问用户 return 'ask'

设计哲学总结

从 1,902 个源文件中提炼出的架构智慧 —— 这些原则适用于所有软件项目

🏎️ 快速路径优先

最常见的操作走最短的路径。--version 不加载任何模块。90% 的性能优化来自于"不做不需要做的事"。

🎣 流式一切

AsyncGenerator 贯穿全栈：API 响应、工具结果、Hook 输出都是流式的。用户永远不用等"全部加载完"才能看到结果。

🛡️ 纵深防御

安全不依赖单一检查。权限系统有 8 层、文件系统有 6 种攻击防御、路径检查覆盖原始路径和符号链接解析后的路径。

🔄 自我纠错闭环

拒绝追踪发现分类器出错时自动退回人工确认。TeammateIdle Hook 防止子代理提前结束。系统能检测并修复自己的问题。

📦 数据驱动 > 硬编码

权限规则是数据、工具是可组合的接口、Hook 是声明式配置。添加新功能不需要改动核心逻辑——只需要添加数据。

🎭 编译时 > 运行时

Feature Gate 在编译时裁剪代码、缓存分界在构建时确定、样式转换在启动时预计算。把工作提前到能提前的最早时刻。

💡 写给编程新手的一句话

优秀的架构不是让代码变"复杂"，而是让复杂的系统变得可理解。

Claude Code 有 1,902 个文件，但每个模块都有清晰的职责边界。工具系统用统一接口让 40+ 工具和谐共处，权限系统用声明式规则替代了无尽的 if/else，渲染引擎用游戏级优化让终端也能丝滑如奶。

当你开始写自己的项目时，记住这个核心原则：先画蓝图，再砌砖。先定接口，再写实现。先想清楚数据怎么流动，再考虑每个节点怎么处理。
